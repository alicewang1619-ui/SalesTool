# 仓库协作设置指引

> git-master 生成的**手动设置清单**。以下涉及仓库权限 / 敏感设置，需你自己在 GitHub 上操作，git-master 不自动改。

## 一、分支保护（保护 main）

前置：仓库已推到 GitHub，且你对仓库有 admin 权限。

### 方式 A · gh CLI（推荐，可复制粘贴）

```bash
# 把 <OWNER>/<REPO> 换成你的仓库
gh api -X PUT repos/<OWNER>/<REPO>/branches/main/protection \
  -H "Accept: application/vnd.github+json" \
  -f required_pull_request_reviews[dismiss_stale_reviews]=true \
  -f required_pull_request_reviews[required_approving_review_count]=1 \
  -f enforce_admins=false \
  -f restrictions= \
  -f allow_force_pushes=false
```

含义：
- PR 至少 1 人 approve 才能合
- 强制 push 关闭（没人能 force-push 到 main）

> 本项目暂未配置 CI 门禁（根目录未检测到测试技术栈）。等以后加了 CI，再在上面加一行
> `-f required_status_checks[strict]=true -f required_status_checks[contexts][]="CI"`，
> 让 PR 必须过 CI 才能合。

### 方式 B · 网页手动设置

1. 仓库页 → **Settings** → **Branches**
2. **Branch protection rules** → **Add rule**
3. Branch name pattern: `main`
4. 勾选：
   - ☑ Require a pull request before merging → Required approvals: `1`
   - ☑ Do not allow bypassing the above settings（可选，连 admin 也卡）
5. **Create** / **Save changes**

## 二、CI 门禁（当前跳过，以后要加时看这里）

根目录未检测到测试技术栈（无 `package.json` / `requirements.txt` 等），所以没生成 `.github/workflows/ci.yml`。

以后要开 CI：
1. 在项目里配好测试（如 Node 的 `npm test`、Python 的 `pytest`）；
2. 在 `.github/workflows/ci.yml` 加对应 workflow（GitHub Actions），`name: CI`；
3. 回到「一、分支保护」把 `required_status_checks` 那两行加上，让 PR 必须过 CI。

> 注：独立子项目 `项目知识库/app` 有自己的 git 仓库，CI 应在它自己的仓库里单独配。

## 三、标签体系建议（可选）

Issues → Labels 配这套（颜色自定）：

| 标签 | 用途 |
| :--- | :--- |
| `needs-triage` | 新提的，待负责人分类（issue 模板默认带） |
| `bug` | 报 Bug 模板默认带 |
| `feature` | 新功能 |
| `enhancement` | 改进现有 |
| `docs` | 文档改动 |
| `priority:high` / `priority:low` | 优先级 |
