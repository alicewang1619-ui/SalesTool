# 场景锚点（V1）· 项目灯塔（Beacon）

**产品名（暂定）**：项目灯塔（Beacon）

**一句话定位**：面向研发团队的多项目统管工具，在一个地方看清所有项目的进度、风险和人力负载。

**核心用户**：
- 管理层 / PMO —— 看全公司项目健康度全局
- 项目负责人 / PM —— 管自己项目、排人、登记风险
- 研发成员 —— 更新自己任务进度
- 不是谁：不对外售卖给其他公司（非 SaaS）；不服务非研发的纯职能团队（第一版聚焦研发）

**核心场景**：150 人研发团队、约 30 个项目并行。当前靠 Excel + 钉钉/微信群 + 脑子管，项目一多，进度、风险、谁忙谁闲全散了，管理层没有一个地方能"一眼看清全局健康度"。

**核心痛点**：
- 进度不透明：30 个项目走到哪、谁拖了，要一个个问
- 风险靠吼：风险散在群聊里，逾期/停滞没人系统盯
- 人力黑盒：不知道谁被排爆了、谁还有余量，资源冲突靠撞

**成功定义**（量化指标待阶段 2 与用户确认）：
- 管理层打开一个看板，≤1 分钟掌握 30 个项目的整体健康度（红/黄/绿）
- 风险/逾期项从"事后才知道"变成"系统自动预警"
- 排人前能先看人力负载，减少资源冲突

**功能边界**：
- In scope（MVP）：多项目总览看板 / 项目进度（里程碑+任务%+甘特）/ 风险登记+自动预警 / 人员分配+人力负载视图
- Out scope（后置）：钉钉/飞书/Jira 集成、SSO、精细到小时的排班日历、移动端完整功能
- 未来再说：跨项目报表、成本核算、客户外部可见

**资源约束**：
- 端：Web 为主 + 手机轻量查看（看进度/审批）
- 部署：公司内网 / 私有服务器，账号自建
- 录入模式：全员协作，人人更新自己任务
- 技术栈：按通用最佳实践由 AI 推荐
- 节奏：无硬截止，先 MVP 分期迭代

**最大不确定性**：
- 全员协作的"没人愿意填"风险 —— 数据新鲜度全靠成员主动更新，推不动则全盘失效
- "项目健康度红黄绿"的判定规则（什么算红）还需与用户细化

**最不该做的方向（防跑偏）**：
- 不要做成"又一个重型 Jira/Worktile"——堆满任务管理细节、要求全员每天精细填工时。灯塔的核心是"管理层一眼看清全局 + 自动预警"，不是替代研发的任务管理工具。往"全功能项目管理平台"做就是死路。

## 假设清单 v1
- A-001：用户主要痛点是"多项目全局不透明"，而非"单项目任务管理不好用"（依据 B：用户原话三件事=进度/风险/人员，均为跨项目统管）
- A-002：全员愿意更新自己任务进度（依据 D：用户选了全员协作，但能否推动未验证，**最大风险**）
- A-003：150 人/30 项目规模下，常规 Web 架构性能足够，无需大数据架构（依据 A：规模明确，量级不大）
- A-004：内网部署、内部工具，无等保/数据出境等特殊合规硬要求，常规权限+操作审计即可（依据 C：内部研发工具惯例，**待用户确认**）
- A-005：项目"健康度"可由 进度偏差 + 逾期任务数 + 风险等级 + 人力超载 等信号自动合成红黄绿（依据 C：通用 PM 实践，规则待细化）
- A-006：手机端只读核心场景（看进度/收预警/审批）即可满足，无需完整录入（依据 B：用户选 Web 为主手机轻量）
- A-007：项目周报 MVP 通过"上传/粘贴文本"喂入灯塔，由系统提取进度/风险/阻塞预填，PM 确认（依据 C：用户告知每个项目都有周报；周报现存形态/位置——钉钉/Word/邮件——待 Stage 5 细化，MVP 默认文本上传以保持独立版不依赖外部系统）
