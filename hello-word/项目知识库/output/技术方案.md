# 技术方案 · 项目知识库

> 锚定 MVP（阶段一八项功能）；local-first；版本号经 `npm view` 实测（2026-06-29）。

## 1. 方案总览
一个**本地优先的桌面应用**：界面用 Web 技术栈，打包成桌面壳，数据与索引全部存本机，大模型本地优先、云端可选。无服务器、无多租户、无登录（单用户自用）。

核心闭环：录入 → 解析/抓取 → 大模型自动整理（标签/摘要）+ 生成向量 → 本地存储 → 语义检索 / 相关推荐 / RAG 问答。

## 2. 整体架构

```text
┌─────────────────────────── 桌面应用（本机） ───────────────────────────┐
│  前端 UI (React + TS)                                                  │
│     │  IPC / 本地调用                                                  │
│  应用核心层（Rust / 或本地服务）                                        │
│     ├── 录入与解析：网页/公众号正文抓取、PDF/Word/MD 解析               │
│     ├── 整理：调用大模型打标签/摘要                                     │
│     ├── 索引：文本分块 + embedding → 写入向量库                         │
│     ├── 检索：query embedding → 向量检索 + 标签/关键词辅助              │
│     └── 问答：检索 Top-K → 拼上下文 → 大模型生成答案 + 来源             │
│  存储层：SQLite（结构化）+ LanceDB（向量），均为本机文件                │
│  模型层：Ollama(本地) ←默认 / 云端 OpenAI 兼容 API(可选)               │
└────────────────────────────────────────────────────────────────────┘
```

## 3. 技术选型（多方案对比 + 推荐）

### 3.1 应用形态：Tauri vs Electron
| | Tauri 2 | Electron |
|---|---|---|
| 安装包 | 小（系统 WebView，几 MB~十几 MB） | 大（捆绑 Chromium，100MB+） |
| 内存 | 低 | 高 |
| 后端语言 | Rust（性能好，门槛略高） | Node（生态熟） |
| AI 编码助手友好度 | 良 | 优（JS 生态） |

> **推荐 Tauri 2**（`@tauri-apps/cli` **2.11.4** / `@tauri-apps/api` **2.11.1**）：local-first 桌面工具追求轻量、低占用，Tauri 完美契合。若团队更熟 JS、想省 Rust 成本，Electron 是稳妥备选。

### 3.2 前端
- **React 19.2.7** + **TypeScript 6.0.3** + **Vite 8.1.0** + **Tailwind CSS 4.3.1**。
- 路由 **react-router-dom 7.18.0**。UI 组件按 tokens.css 自建/配 shadcn 风格。

### 3.3 存储
- 结构化：**SQLite**（Tauri 用 `tauri-plugin-sql` 或 Rust `rusqlite`；Node 方案用 **better-sqlite3 12.11.1**）。
- 向量：**@lancedb/lancedb 0.30.0**（嵌入式、列式、零外部服务，本地优先首选）。备选 **chromadb 3.4.3**（上手快，但更偏服务化）。

### 3.4 大模型 / RAG
- 本地：**ollama 0.6.3**（Node SDK）驱动本地模型（如 qwen2.5）。
- RAG 编排：可用 **@langchain/core 1.2.1** 或自研轻量编排（几千条规模，自研更可控、依赖更少——**推荐自研轻量 RAG**）。
- embedding：本地中文友好模型（bge-small-zh 类，经 Ollama 或本地推理）。

## 4. 数据与集成
- **核心实体**：Knowledge / Tag / KnowledgeTag / Embedding（见 PRD §5）。
- **正文抓取**：通用网页用 Readability 类算法；微信公众号单独适配。**抓取失败回退粘贴**（应对反爬，A-001）。
- **文件解析**：PDF（pdf 解析库）、Word（docx 解析）、Markdown（直接读）。
- **数据访问**：前端经 IPC/本地调用访问本机 SQLite + LanceDB，无网络请求（除非调云端模型）。
- **同步**：MVP 无云同步；提供导出/导入备份。

## 5. 大模型与 RAG 细节
- **本地优先**：默认走 Ollama，数据不出本机、可离线。
- **云端可选**：配置 OpenAI 兼容 API，**国内可达硬约束** → 推荐 DeepSeek / 通义千问 / 智谱 GLM 等国内可稳定访问的服务（不默认推荐需翻墙的海外直连端点）。
- **RAG 流程**：分块 → embedding → 向量检索 Top-K → 拼上下文 → 生成 → 标注来源 knowledge_id。
- **一致性**：正文更新 → 重建该条 embedding；删除 → 级联删向量。

## 6. 安全与隐私
- 数据默认全本地，不上传。
- 用云端模型时仅发送必要上下文，并在 UI 明确提示（设置页隐私条）。
- 单用户本地，MVP 默认免登录；可选本地密码锁（后置）。
- 删除走软删 + 回收站，防误删（AC-8 相关）。

## 7. 部署与运维
- 打包为各平台桌面安装包（Tauri 出 .dmg/.msi/.AppImage）或一键启动脚本。
- 本地数据可一键导出备份 / 从备份导入恢复。
- 个人自用：审计日志轻量或不做。

## 8. 风险与里程碑
| 风险 | 缓解 |
|---|---|
| 公众号反爬抓不到正文（A-001） | 通用解析 + 公众号适配；失败回退粘贴 |
| 低配机跑本地大模型慢/效果差（A-002） | 云端兜底；embedding 用轻量中文模型；几千条规模检索压力小 |
| Rust 上手成本 | 团队不熟则退 Electron + Node |

**里程碑（建议）**：M1 录入+存储+列表（FR-1/FR-8）→ M2 自动整理+向量索引（FR-2/FR-7）→ M3 语义检索+相关推荐+标签（FR-3/FR-4/FR-5）→ M4 RAG 问答+设置/备份（FR-6/FR-8）。

> 先验证两条假设：公众号抓取可行性、目标机本地大模型性能（PRD OQ-1/OQ-2）。
