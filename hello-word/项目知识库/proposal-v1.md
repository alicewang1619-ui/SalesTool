# 方案 V1 · 项目知识库

## 一、产品一句话
本地优先的个人 AI 知识库，聚焦中文内容生态（公众号/中文 AI 对话）与开箱即用，把多来源信息沉淀为可语义检索、可关联、可问答的个人知识资产。

## 二、核心闭环
```text
丢进来 → 自动整理 → 找得回 → 看关联
(录入)  (标签+摘要)  (语义检索)  (相关推荐+标签)
            ↘ 基于库问答(RAG) ↙
```

## 三、功能清单（带优先级）

| 编号 | 功能 | 说明 | 优先级 |
|---|---|---|---|
| F1 | 内容录入 | 贴链接抓正文 / 复制粘贴 / 编辑器写 / 上传文件 | P0 |
| F2 | 自动整理 | 大模型自动打标签 + 生成摘要 + 归类 | P0 |
| F3 | 语义检索 | 自然语言按意思搜 | P0 |
| F4 | 相关推荐 | 看某条时自动列出相关知识 | P0 |
| F5 | 标签聚合 | 按标签浏览/筛选 | P1 |
| F6 | 库内问答(RAG) | 基于个人库提问，AI 找答案并总结、标来源 | P0/P1（待拍板） |
| F7 | 大模型接入 | 本地优先(Ollama)，公开大模型可选(API) | P0（底座） |
| F8 | 知识管理 | 列表/详情/编辑/删除 | P0 |

## 四、技术方案（AI 按 local-first 最佳实践推荐，待技术评审细化）
- 形态：本地运行的应用——桌面壳(Tauri/Electron) 或 本地服务 + 浏览器访问 localhost
- 本地大模型/embedding：Ollama（用户已确认本地优先）
- 本地向量库：LanceDB / Chroma（嵌入式、无需独立服务）
- 正文抓取：公众号/网页正文解析（Readability 类 + 公众号适配）
- 数据：本地文件 + 本地向量索引，数据全在本机
- 云端大模型：可配置 API Key，按需切换

## 五、MVP 划分草案（待企业家阶段 4.6 拍板）
- **阶段一 MVP（核心闭环必含）**：F1(录入：贴链接+粘贴+编辑器写) + F2(自动整理) + F3(语义检索) + F4(相关推荐) + F7(本地+云端大模型) + F8(知识管理)
- **待拍板**：F6 库内问答(RAG) 是否进 MVP（核心卖点之一，但开发量不小）；F1 的"上传文件"是否进 MVP
- **完成定义**：能把一篇公众号/一段 AI 对话丢进去，系统自动打标签摘要，之后用大白话搜到它并看到相关知识
- **后置**：浏览器插件、手机端、多端云同步、多人/多岗位共用、关系图谱、公众号抓取深度优化

## 六、四视角风险自审

| 视角 | 风险（失败模式） | 缓解 |
|---|---|---|
| 技术 | 微信公众号反爬，链接抓不到正文(A-001) | 抓取失败自动回退"粘贴正文"；先支持通用网页+公众号尽力而为 |
| 技术 | 低配电脑跑本地 LLM/embedding 慢或效果差(A-002) | 支持云端大模型兜底；embedding 用轻量中文模型；几千条规模向量检索压力小 |
| 设计 | 录入太重导致不坚持用(A-003，成败关键) | 贴链接一键入库、粘贴即存、最少必填；自动整理省去手动归类 |
| 商业/资源 | 自研工作量超出 AI 编码助手一次能搞定的范围 | MVP 砍到最小闭环先跑通，F6/文件上传按拍板决定是否后置 |
| 战略 | 与 Reor 差异化不足，沦为重复造轮子 | 死守中文+公众号+中文AI对话+开箱即用；数据代码完全自有 |
