# 智能销售接待与客户洞察Agent 对话记录

## 2026-06-29 阶段 0 · 自由倾诉

### 用户原始描述

用户做数字化营销，目前的问题是：

- 询盘来源很散。
- 线上沟通人工效率低、不及时。
- 询盘整理很费时间。
- 给销售分发询盘信息靠人工微信或邮件。
- 搜集销售反馈很难。
- 希望有一个 Agent 帮忙处理这些事情。

### 用户补充 1

用户说明：不仅仅是询盘，还要加上客户调研和分析、背景查询，完成客户调研工作。

### 用户补充 2

用户说明：在线沟通环节也很重要。Agent 需要向客户讲产品并进行引导式提问。

示例：

- 客户说："Hi"
- Agent 应回复："What can I help you?"
- 客户说："I wanted to buy an ultrasound"
- Agent 应继续询问："For what application?"

用户强调还有很多类似的细节引导话语。

### 用户补充 3

用户说明：销售线索调度分发给销售后，还要追踪销售反馈，并且每天形成一个询盘报告发给用户。

### 当前理解

产品不是单纯客服机器人，也不是单纯询盘管理系统，而是一个覆盖售前接待、客户背景调研、销售线索调度、销售反馈追踪和每日汇报的智能销售接待与客户洞察 Agent。

### 用户问题：需求完整度与竞品做法

用户询问：这个需求是否完整，以及竞品或相关产品通常怎么做。

待回应重点：当前方向完整，但还不是可开发 PRD；成熟产品通常拆成在线接待/AI SDR、客户数据补全、线索评分、实时路由、CRM 跟进和管理报表几层。

### 用户补充：客户分层与再营销

用户补充：客户信息收集后，后续应按照不同销售进度和客户类别，进行再次营销沟通，例如邮件群发。

产品范围更新：在原有“接待 + 背调 + 分发 + 反馈 + 日报”之外，增加“客户分层与再营销培育”。该能力需要后续确认客户阶段定义、客户类别、触达渠道、触达频率、内容模板、退订/合规边界和效果回流。

### 用户补充：客户动态监测

用户补充：后续如果已经掌握客户的网站、LinkedIn、Facebook 等信息，可以对这些平台进行监测；当出现值得跟进的动态时，可以给客户发信息。

产品范围更新：增加“客户动态监测与机会触发”。该能力需要后续确认监测数据源、授权方式、监测频率、机会信号定义、自动触达边界、人工审核机制、退订/合规约束和触达效果回流。

### 用户澄清：客户态势监督

用户澄清：严格来说，这一能力应该算“客户情况监督”，用于发现商机以及进行客户背景调查，而不是单纯的社媒或平台动态监测。

术语更新：将“客户动态监测”调整为“客户态势监督与商机发现”。该模块覆盖客户背景调查、持续观察客户变化、识别商机信号、生成跟进建议，并可为后续再营销提供触发依据。

### 用户回答：目标客户类型

用户澄清：虽然系统是 B2B 销售增长系统，但客户群体不仅是 B2B 企业。实际面对的客户包括：

2. 终端用户，尤其是医生类型

产品判断：后续 AI 接待话术、客户背景调查和线索评分不能只按 B2B 企业设计，必须区分“代理商线索”和“医生/终端用户线索”两套判断逻辑。

### 用户回答：高质量线索标准

用户定义高质量线索的核心标准：这个人能明确说出自己的身份，并且给出具体业务需求。

具体要求：

1. 明确的身份信息
   - 所在机构类型，例如 Company、Hospital、Doctor
   - 具体应用领域，例如 Radiology、Cardiology
2. 明确的需求描述
   - 必须给出具体业务需求
   - 只有这样才能确定这是准确询盘，而不是泛泛咨询或简单询价

产品判断：线索评分的硬门槛应是“身份明确 + 应用明确 + 需求明确”。Agent 的关键任务之一，是通过引导提问把模糊询盘补全到这三个条件。

### 用户补充：代理商与医生画像字段

对于代理商，需要关注：

1. 重点国家
2. 历史合作业务：之前代理过哪些品牌的机器，或者做过什么样的业务，例如 IVD、MR 等
3. 超声经验：现在有没有做过超声；如果做过，涉及哪些品牌
4. 业务规模：目前一年的业务量大概是多少

对于医生，需要关注：

1. 基本信息：所属科室及具体临床应用
2. 使用习惯：之前用过什么样的机器
3. 财务状况：预算情况
4. 核心需求：想购买什么样的产品，打算应用在什么场景

产品判断：客户画像与线索评分应分为“代理商画像”和“医生/终端用户画像”两套字段。Agent 的询问与客户背景调查应根据客户身份动态切换。

### 用户回答：销售分发规则

用户确认销售分发原则：按照国家区域划分。

具体逻辑：

2. 在此前提下，进一步确认具体国家
3. 根据不同国家，将业务分配给对应销售人员

产品判断：线索路由第一优先级为“国家/区域”。系统需要维护区域-国家-销售人员映射表，Agent 在客户信息不完整时应优先补齐国家。

### 用户回答：销售反馈状态与现状

用户确认销售应反馈的状态包括：

3. 确认是真实需求
4. 已转给代理商跟进（通常会给代理商去跟进）

当前沟通方式：微信和邮件。当前没有 CRM 系统。用户担心让销售填简单表单会让销售觉得麻烦。

产品判断：不应一开始建设重 CRM，也不应要求销售重复录入客户信息。更合适的方案是“微信/邮件通知 + 一键反馈卡片 + 内置轻量线索看板”。系统预填客户摘要、AI 建议和状态选项，销售只需点击状态并可选填一句备注。

### 用户确认：销售反馈必须轻量

用户确认：如果做 CRM 会太复杂，沟通方式应让销售用最简单、最方便的方式反馈。销售工作繁忙，无法花太长时间写反馈；同时销售的一线判断比系统更了解真实客户。

产品原则：

1. AI 负责整理、预填、提醒、汇总
2. 销售只做最小动作反馈，例如点状态、选判断、补一句备注
3. 系统不强迫销售重复录入客户信息
4. 销售判断优先于 AI 判断，AI 只能辅助和提醒
5. 反馈交互应控制在 10-30 秒内完成

### 用户回答：客户分析报表指标

用户每天需要看：

1. 总询盘数
2. 有效询盘数
3. 不同国家/区域的询盘分量

需要分析的维度：

1. 每个销售的反馈以及不同渠道的线索分析
   - 对比 Facebook 和官网的有效客户数量
   - 了解不同区域销售的反馈情况，明确哪些区域反馈较多，哪些区域不反馈
2. 跟进后的转化分析
   - 哪个区域的销售跟进转化较多
   - 哪个来源渠道的询盘转化率更高
3. 产品分析
   - 在所有询盘中统计不同产品的询盘分布，例如 Sonos Max、Sonos I
   - 哪个产品询盘更多，哪款产品转化率更高

产品判断：报表体系至少包含四类核心视图：总量漏斗、渠道质量、区域/销售反馈、产品需求与转化。
