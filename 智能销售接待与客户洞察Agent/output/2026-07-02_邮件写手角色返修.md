# 2026-07-02 邮件写手角色返修

## PRD 口径
- 邮件写手角色不再维护或展示中文名；后台配置与前端下拉统一使用英文角色名，例如 `ReplyMirror`、`Baymax`、`Mario`。
- 邮件写手角色必须支持配置：能力与技能方向、角色目标、相关技能、其他背景定义、角色定义标签、风格、适用场景和启用状态。
- 新增默认角色 `ReplyMirror`，定位为“回邮镜”：用于复盘客户来信、镜像客户真实需求、保持自然回邮语气，并在合规边界内推动下一步回复。
- 再营销草稿的大模型输入必须包含邮件目的、选中写手完整角色定义、客户摘要、背景调查、销售反馈、建议下一步动作、人工提示词和参考附件元数据。
- 写邮件页面必须展示一个“小标签/字段”提示本次发邮件目的；目的和角色定义一起进入 `prompt_context_snapshot`。
- 当用户更换写手角色、修改发邮件目的或点击重新生成时，草稿必须重新基于当前角色 Prompt 生成，不得继续沿用旧角色生成的模板。

## 设计口径
- 设置页“邮件写手角色”弹窗字段顺序：英文角色名、角色 Key、能力与技能方向、角色目标、相关技能、背景定义、角色定义标签、适用场景、启用状态。
- 设置页角色摘要卡展示英文角色名、风格、目标、能力方向和标签；不展示“中文名”字段。
- 草稿详情页在“邮件写手角色”附近展示邮件目的标签，例如 `Purpose: Post-quote follow-up`，并允许用户调整目的后触发重新生成。

## 测试口径
- 保存邮件写手配置后，`/api/ai/email-writers` 返回英文角色名和完整角色定义字段，不依赖中文名。
- 再营销重新生成后，`prompt_context_snapshot` 必须包含 `email_purpose`、`writer_role_goal`、`writer_role_capabilities`、`writer_role_background`、`writer_role_tags`。
- 切换不同写手角色后，`draft_content` 和上下文快照必须同步变化，证明草稿来自当前角色定义而不是旧模板。

## 实现回写
- 后端 `NurtureTask` 新增 `email_purpose`，SQLite 兼容迁移默认填充 `Follow-up reply`。
- 后端邮件写手角色保留 `display_name` 兼容旧接口，但设置保存与前端展示均以英文 `name` 为准；旧默认写手的中文风格/技能会归一化为英文默认配置。
- 后端生成上下文已加入邮件目的、角色能力、角色目标、背景定义和标签；真实模型不可用时，英文兜底模板也会根据角色标签与邮件目的生成不同正文。
- 前端草稿详情页新增邮件目的字段和 `Purpose` 标签；切换邮件目的或写手角色会直接调用重新生成接口，避免旧模板残留。
- 设置页邮件写手弹窗移除中文名字段，新增能力与技能方向、角色目标、相关技能、背景定义和角色定义标签。
