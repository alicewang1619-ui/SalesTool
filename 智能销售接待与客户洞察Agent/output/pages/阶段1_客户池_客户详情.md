# 客户详情

## 1. 文档元信息

| 字段 | 内容 |
| :--- | :--- |
| 所属产品 | 智能销售接待与客户洞察 Agent |
| 所属模块 | 客户池 |
| 适用端 | Web |
| 页面路由 | `/admin/customers/:customerId` |
| 页面类型 | 详情 |
| 所属阶段 | 阶段1(MVP) |
| 上游来源 | 客户池列表 / 线索池列表 / 工作台 |
| 下游去向 | 草稿详情确认发送、客户态势信号记录 |

## 2. 页面概述

### 2.1 一句话定位
作为唯一客户详情页，聚合客户历史线索、客户背景调查、反馈、状态变化、再营销记录和态势信号；从线索池或客户池进入都使用同一页面。

### 2.2 存在理由
1. 没有该页面，线索闭环中的对应动作会回到人工微信、邮件或表格。
2. 没有统一入口，管理者无法判断客户背景、线索状态、销售反馈和客户池变化。
3. 该页面把 PRD 中“少填、快看、可追溯”的要求落为具体操作。

### 2.3 页面目标（可衡量）

| 目标维度 | 具体指标 |
| :--- | :--- |
| 效率 | 关键操作在 3 次点击内完成 |
| 准确性 | 状态、负责人、来源和时间字段完整展示 |
| 可追溯 | 客户背景调查来源、人工修改、导出、发送或提交均能回到记录 |

### 2.4 不做什么
- 不展示成交金额和报价金额，因为 PRD 明确排除金额类指标。
- 不做完整 CRM 的复杂商机阶段管理，第一版聚焦线索闭环。
- 不做未授权 Facebook/LinkedIn 抓取。

## 3. 用户与场景

### 3.1 目标用户

| 维度 | 描述 |
| :--- | :--- |
| 角色 | 管理员、营销/运营、销售，按权限看到不同动作 |
| 技术熟练度 | 中等，熟悉表格和基础后台操作 |
| 使用设备 | PC 浏览器为主 |
| 使用姿势 | 高频扫描、快速判断、少量确认 |

### 3.2 核心使用场景

**场景 A：正常处理**
- 时机：用户进入 客户详情 执行日常任务。
- 前置：已登录或持有有效安全链接。
- 动作：查看客户背景调查、关键信息和历史记录，完成筛选、确认、反馈、导入或查看详情。
- 情绪：清楚、稳定、知道下一步。

**场景 C：人工修正客户背景调查**
- 时机：系统自动生成的客户背景调查不完整，或销售/运营掌握了新的客户信息。
- 前置：当前用户具备客户背景编辑权限。
- 动作：编辑调查摘要、补充来源说明、保存人工修改，系统记录操作人和修改时间。
- 情绪：可控、可追溯，不担心 AI 结论覆盖人工判断。

**场景 B：异常兜底**
- 时机：数据缺失、链接过期、权限不足或任务失败。
- 前置：当前流程无法自动完成。
- 动作：查看原因，进入待处理、重试、返回工作台或请求管理员处理。
- 情绪：不被卡住，知道谁来处理。

### 3.3 用户核心诉求
1. 快速看到状态和下一步。
2. 不重复录入系统已有信息。
3. 能追溯 AI 判断、销售反馈和人工修改来源。

## 4. 信息架构

### 4.1 区域划分
1. 顶部标题与关键动作。
2. 核心指标或摘要区。
3. 折叠模块区：AI 评分、建议动作/再营销、背景调查、态势信号、历史记录。
4. 异常提示和权限说明区。

### 4.2 区域内容
- 标题区展示页面名称、当前状态和主操作。
- 从线索池进入时，标题区用辅助文本展示来源线索，并在历史线索表中高亮该条来源记录。
- 主体区展示与 客户池 相关的核心业务数据。
- 折叠模块区默认展开 AI 评分与建议动作/再营销；客户背景调查、态势信号和历史记录默认收起。
- 客户背景调查折叠面板展示系统自动生成的背景结论、来源证据、可信度、最近更新时间，并提供人工编辑入口。
- 建议动作折叠面板展示该客户对应的下一步建议、再营销草稿入口、邮件状态和参考附件数量；只允许展示当前客户自己的建议动作。
- 历史记录折叠面板展示历史线索、销售反馈和客户时间线。

## 5. 界面内容

### 5.1 元素清单
- 页面标题：客户详情。
- 来源线索提示：从线索池进入时展示来源线索、反馈状态和进入时间。
- 客户背景调查：自动调查结论、官网/邮箱/历史邮件/人工补充来源、可信度、更新时间、编辑文本框。
- 建议动作：推荐下一步动作、客户备注、审批状态、邮件草稿状态、参考附件数量、查看邮件草稿入口。
- 折叠面板：AI 评分、建议动作/再营销、客户背景调查、客户态势信号、历史记录；标题带评分、数量或状态标签。
- 状态标签：有效、未反馈、待分配、待确认、已完成等。
- 筛选或表单控件：来源、国家、客户类型、产品、销售、周期。
- 主操作按钮：查看详情、确认、保存、导入、导出或提交反馈。
- 记录列表：至少展示客户名称、国家、渠道、产品、负责人、更新时间。

### 5.2 辅助信息
- AI 判断理由、客户背景调查来源、数据来源、更新时间、导出脱敏说明或链接有效期说明。

## 6. 交互流程

### 6.1 页面加载与初始化
- 校验登录态或安全链接。
- 拉取页面所需数据。
- 按业务时间倒序展示列表或记录。

### 6.2 核心操作流程

```text
进入客户详情
  -> 查看关键状态和上下文
  -> 如来自线索池，确认来源线索上下文
  -> 查看客户背景调查及来源证据
  -> 查看该客户建议动作，必要时进入邮件草稿
  -> 执行主操作
  -> 系统保存结果并记录来源
  -> 跳转或回写到下游页面：草稿详情确认发送、客户态势信号记录
```

### 6.3 客户背景调查修改流程

```text
进入客户详情
  -> 打开客户背景调查区
  -> 查看系统自动生成结论和来源证据
  -> 人工编辑调查文本或补充来源
  -> 保存
  -> 系统记录修改人、修改时间、修改前后内容和来源
```

### 6.4 辅助操作
- 搜索、筛选、排序、重试、返回上级和导出确认。

## 7. 状态与边界

### 7.1 页面状态全景

| 状态 | 触发条件 | 页面表现 |
| :--- | :--- | :--- |
| 初始态 | 首次进入 | 展示标题、骨架或默认数据 |
| 加载态 | 请求中 | 骨架屏或按钮 loading |
| 完整态 | 数据可用 | 展示 mock 数据和关键动作 |
| 空态 | 无数据 | 给出导入、配置或返回入口 |
| 失败态 | 请求失败 | 展示原因和重试 |
| 无权限 | 角色不允许 | 隐藏或置灰动作并说明原因 |

### 7.2 边界与容错

| 边界场景 | 处理策略 |
| :--- | :--- |
| 网络异常 | 保留当前输入，允许重试 |
| 会话过期 | 跳回登录页或提示重新发送链接 |
| 连续错误 | 显示可理解原因，不吞掉错误 |
| 异常输入 | 当前字段就地提示 |
| 并发操作 | 以最新保存记录为准并保留操作人 |
| 权限不足 | 动作置灰或进入无权限页 |
| 背景调查生成失败 | 展示失败原因，允许重新生成或人工录入 |
| 来源可信度低 | 标记待复核，不作为自动评分唯一依据 |
| 人工修改与 AI 结论冲突 | 保留两者，业务展示以人工修改为准 |

## 8. 业务规则与校验

### 8.1 字段定义

| 字段 | 说明 |
| :--- | :--- |
| source_channel | 官网聊天、官网后台、邮箱、展会名片等来源 |
| country | 决定区域和销售负责人 |
| customer_type | 代理商、Doctor、Hospital、Clinic 等 |
| product_interest | Trolley、Portable、Handheld 或具体型号 |
| feedback_status | 已联系、已报价、无效、需跟进、已成交 |
| background_investigation | 客户背景调查正文 |
| background_sources | 官网公开信息、邮箱域名、历史邮件、销售反馈、人工补充等来源 |
| background_confidence | 背景调查可信度和待复核状态 |

### 8.2 校验规则

| 校验项 | 时机 | 规则 | 失败反馈 |
| :--- | :--- | :--- | :--- |
| 必填项 | 提交时 | 关键字段不能为空 | 标记待补充 |
| 权限 | 操作前 | 角色必须允许动作 | 动作不可用 |
| 链接有效期 | 打开反馈卡片 | 7 天内有效 | 提示重新发送 |
| 导出字段 | 导出前 | 不包含成交金额和报价金额 | 阻止导出 |
| 背景调查编辑 | 保存时 | 必须有修改权限，保存时记录修改原因或来源 | 阻止保存并提示权限/字段问题 |
| 建议动作归属 | 渲染客户详情时 | 只展示 `customer_id` 与当前客户一致的 NurtureTask | 不展示建议动作，避免把别的客户待办误放入当前客户详情 |

### 8.3 特殊业务规则
- 销售判断优先于 AI 评分，但 AI 判断理由保留。
- 客户背景调查由系统自动生成，人工修改优先于自动结论；系统保留原始自动版本和来源证据。
- 背景调查只使用客户官网公开信息、邮箱域名、授权邮件互动、销售反馈和人工补充，不做未授权社媒抓取。
- 再营销必须人工确认后发送。
- 官网 KPI 只统计渠道归因和时效，不统计金额。

## 10. 数据需求

### 10.1 页面需要什么数据

| 数据 | 来源 | 用途 |
| :--- | :--- | :--- |
| 当前用户与角色 | Auth 模块 | 权限控制 |
| 线索/客户/反馈记录 | 后端业务模块 | 展示主体内容 |
| 客户背景调查 | BackgroundInvestigation / CustomerProfile | 展示自动调查结论、来源证据和人工修改记录 |
| 配置字典 | 系统配置 | 渲染国家、产品、销售选项 |
| 报表聚合 | ReportMetric | 展示指标 |

### 10.2 页面提交什么数据

| 数据 | 触发时机 | 用途 |
| :--- | :--- | :--- |
| 筛选条件 | 查询时 | 获取列表 |
| 状态或配置修改 | 保存时 | 更新业务记录 |
| 客户背景调查人工修改 | 保存背景调查时 | 更新客户背景调查版本并写入审计 |
| 反馈或确认结果 | 提交时 | 写入闭环 |

### 10.3 明确排除的数据
- 成交金额、报价金额。
- 未授权社媒私域数据。
- 与当前页面无关的完整 CRM 字段。

## 12. 生产化落地补充（06-30-2026）

- 客户详情真实路由为 `/admin/customers/:customerId`，对应接口为 `GET /api/customers/{customer_id}`。
- 页面必须展示全局 Banner、客户基础摘要、客户分层、负责人、历史线索数量、反馈记录数量。
- “客户背景调查”模块展示自动调查原文、当前生效摘要、来源证据、可信度、更新时间和更新人；有权限用户可在同一区域人工修订并保存。
- 来源证据以结构化表格展示，字段为来源类型、标题和证据内容，来源可以来自官网公开信息、邮箱域名/历史邮件、销售反馈和人工补充。
- 详情页同时展示历史线索、销售反馈记录和状态时间线，保证背景调查、线索闭环和销售反馈可追溯。
- 详情页展示“建议动作”模块，用于承接客户池列表不再展示的客户级下一步动作；该模块只能展示当前客户自己的待确认再营销任务，并提供进入邮件草稿的链接。
- 管理员和运营可编辑客户背景调查；销售只可查看自己负责客户的详情，保存按钮必须不可用，后端写入接口继续返回 403。
- 保存人工修订时，后端必须保留自动调查原文，只更新人工版本、更新人、更新时间，并写入 `update_customer_background` 审计日志。
- 前端加载态只展示加载卡片；客户背景调查编辑表单在详情数据返回后挂载，并以当前背景调查版本作为初始值，避免空数据状态下误保存或触发表单告警。
- 前端必须对 `background.sources`、`lead_history`、`feedback_records`、`timeline`、`signals` 做运行时缺省保护；当旧接口、导入客户或存量客户暂未返回这些数组时，页面展示 0、空表格或“暂无背景调查”，不得出现 `Cannot read properties of undefined` 崩溃页。

- 生产实现以 React/Vite 路由和 FastAPI 真实 API 为准，HTML 文件仅作为视觉、布局和组件蓝图，不作为可执行生产代码。

## 2026-07-02 全局流程返修补充
