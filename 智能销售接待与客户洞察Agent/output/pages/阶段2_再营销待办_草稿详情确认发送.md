| 字段 | 内容 |
| :--- | :--- |

## 2. 页面概述

### 2.1 一句话定位
查看建议下一步动作和客户备注，编辑大模型生成的邮件草稿，补充生成提示词和附件素材，确认发送、取消草稿和查看发送结果。

### 2.2 存在理由
1. 没有该页面，线索闭环中的对应动作会回到人工微信、邮件或表格。
2. 没有统一入口，管理者无法判断线索状态、销售反馈和客户池变化。
3. 该页面把 PRD 中“少填、快看、可追溯”的要求落为具体操作。

### 2.3 页面目标（可衡量）

| 目标维度 | 具体指标 |
| :--- | :--- |
| 效率 | 关键操作在 3 次点击内完成 |
| 准确性 | 状态、负责人、来源和时间字段完整展示 |
| 可追溯 | 人工修改、导出、发送或提交均能回到记录 |

### 2.4 不做什么
- 不展示成交金额和报价金额，因为 PRD 明确排除金额类指标。
- 不做完整 CRM 的复杂商机阶段管理，第一版聚焦线索闭环。
- 不做未授权 Facebook/LinkedIn 抓取。

## 3. 用户与场景

### 3.1 目标用户

| 维度 | 描述 |
| :--- | :--- |
| 角色 | 管理员、营销/运营、销售，按权限看到不同动作 |
| 技术熟练度 | 中等，熟悉表格和基础后台操作 |
| 使用设备 | PC 浏览器为主 |
| 使用姿势 | 高频扫描、快速判断、少量确认 |

### 3.2 核心使用场景

**场景 A：正常处理**
- 时机：用户进入 草稿详情确认发送 执行日常任务。
- 前置：已登录或持有有效安全链接。
- 动作：查看关键信息，完成筛选、确认、反馈、导入或查看详情。
- 情绪：清楚、稳定、知道下一步。

**场景 B：异常兜底**
- 时机：数据缺失、链接过期、权限不足或任务失败。
- 前置：当前流程无法自动完成。
- 动作：查看原因，进入待处理、重试、返回工作台或请求管理员处理。
- 情绪：不被卡住，知道谁来处理。

### 3.3 用户核心诉求
1. 快速看到状态和下一步。
2. 不重复录入系统已有信息。
3. 能追溯 AI 判断、销售反馈和人工修改来源。

## 4. 信息架构

### 4.1 区域划分
1. 顶部标题与关键动作。
2. 核心指标或摘要区。
3. 主操作区。
4. 明细表格、时间线或表单区。
5. 异常提示和权限说明区。

### 4.2 区域内容
- 标题区展示页面名称、阶段、当前状态和主操作。
- 主体区按照“建议下一步动作、客户备注、触达理由、邮件草稿、生成提示词、附件素材”的顺序展示。
- 明细区展示客户摘要、草稿编辑、大模型生成输入、人工确认、取消原因和发送结果。

## 5. 界面内容

### 5.1 元素清单
- 页面标题：草稿详情确认发送。
- 状态标签：有效、未反馈、待分配、待确认、已完成等。
- 表单控件：建议下一步动作、客户备注、触达理由、邮件草稿、生成提示词/补充指令、附件上传、取消原因。
- 主操作按钮：保存修正、人工确认发送、取消草稿、返回客户详情。
- 记录区：展示客户摘要、客户池分层、最近销售反馈、发送审计记录。

### 5.2 辅助信息
- AI 判断理由、数据来源、更新时间、导出脱敏说明或链接有效期说明。

## 6. 交互流程

### 6.1 页面加载与初始化
- 校验登录态或安全链接。
- 拉取页面所需数据。
- 按业务时间倒序展示列表或记录。

### 6.2 核心操作流程

```text
进入草稿详情确认发送
  -> 查看关键状态和上下文
  -> 执行主操作
  -> 系统保存结果并记录来源
  -> 跳转或回写到下游页面：客户详情
```

### 6.3 辅助操作
- 搜索、筛选、排序、重试、返回上级和导出确认。

### 7.1 页面状态全景

| 边界场景 | 处理策略 |

## 8. 业务规则与校验

### 8.1 字段定义

| 字段 | 说明 |
| :--- | :--- |
| source_channel | 官网聊天、官网后台、邮箱、展会名片等来源 |
| country | 决定区域和销售负责人 |
| customer_type | 代理商、Doctor、Hospital、Clinic 等 |
| product_interest | Trolley、Portable、Handheld 或具体型号 |
| feedback_status | 已联系、已报价、无效、需跟进、已成交 |
| recommended_next_action | AI 建议下一步动作，位于客户备注上方，确认前可人工修正 |
| customer_note | 客户池备注或销售反馈备注，展示在建议下一步动作下方 |
| draft_content | 大模型邮件草稿正文，确认发送前必须可编辑 |
| generation_prompt | 本次生成邮件使用的提示词或补充指令，可由运营人工修改 |
| prompt_context_snapshot | 生成时使用的客户摘要、客户背景调查、客户备注、销售反馈和建议下一步动作快照 |
| attachment_refs | 上传附件素材引用，如产品彩页、对比表、应用案例或合规资料 |
| model_provider / model_version | 生成草稿的大模型供应商和模型版本 |
| approval_status | 待确认、已确认、已取消、发送失败 |

### 8.2 校验规则

| 校验项 | 时机 | 规则 | 失败反馈 |
| :--- | :--- | :--- | :--- |
| 必填项 | 提交时 | 关键字段不能为空 | 标记待补充 |
| 权限 | 操作前 | 角色必须允许动作 | 动作不可用 |
| 链接有效期 | 打开反馈卡片 | 7 天内有效 | 提示重新发送 |
| 导出字段 | 导出前 | 不包含成交金额和报价金额 | 阻止导出 |

### 8.3 特殊业务规则
- 销售判断优先于 AI 评分，但 AI 判断理由保留。
- 再营销必须人工确认后发送。
- 确认发送前必须能看到建议下一步动作；动作被人工修改时记录修改人、时间和修改前后内容。
- 邮件草稿由大模型生成，生成输入必须包含客户摘要、客户背景调查、客户备注、销售反馈、人工提示词和附件素材；提示词和附件更新后可重新生成草稿并留痕。
- 邮件草稿生成前必须可选择“邮件写手角色”；角色来自设置页配置，包含英文标识、中文名、风格、技能和适用场景。保存或重新生成后，选中角色进入 NurtureTask，并写入大模型上下文快照。
- 附件素材不得绕过价格、注册证书、独家代理政策等禁用承诺；不合规附件阻止进入发送确认。
- 官网 KPI 只统计渠道归因和时效，不统计金额。

## 10. 数据需求

### 10.1 页面需要什么数据

| 数据 | 来源 | 用途 |
| :--- | :--- | :--- |
| 当前用户与角色 | Auth 模块 | 权限控制 |
| 线索/客户/反馈记录 | 后端业务模块 | 展示主体内容 |

| 报表聚合 | ReportMetric | 展示指标 |

### 10.2 页面提交什么数据

| 数据 | 触发时机 | 用途 |
| :--- | :--- | :--- |
| 筛选条件 | 查询时 | 获取列表 |
| 状态或配置修改 | 保存时 | 更新业务记录 |
| 反馈或确认结果 | 提交时 | 写入闭环 |
| 下一步动作、草稿、提示词和附件修正 | 保存或确认前 | 写入 NurtureTask 修改历史 |

### 10.3 明确排除的数据
- 成交金额、报价金额。
- 未授权社媒私域数据。
- 与当前页面无关的完整 CRM 字段。
