# 客户态势信号记录

## 1. 文档元信息

| 字段 | 内容 |
| :--- | :--- |
| 所属产品 | 智能销售接待与客户洞察 Agent |
| 所属模块 | 客户态势 |
| 适用端 | Web |
| 页面路由 | `/admin/customer-signals` |
| 页面类型 | 列表/表单 |
| 所属阶段 | 阶段2 |
| 上游来源 | 工作台 / 左侧导航 |
| 下游去向 | 客户详情 |

## 2. 页面概述

### 2.1 一句话定位
记录客户官网公开信息、邮件互动、销售反馈和人工录入的机会信号。

### 2.1.1 生产化口径
- 生产实现以 React 路由 `/admin/customer-signals` 和 FastAPI `/api/customer-signals` 为准，HTML 文件仅作为视觉、布局和组件蓝图。
- 页面是管理员/营销运营的全局客户态势视图；销售角色不进入该全局页面，销售仍只在客户详情或反馈卡片中看到本人负责范围。
- 信号来源只允许官网公开信息、授权或导入的邮件互动、销售反馈和人工录入；Facebook/领英只能作为已授权归档或手工来源标签，不触发未授权抓取。

### 2.2 存在理由
1. 没有该页面，线索闭环中的对应动作会回到人工微信、邮件或表格。
2. 没有统一入口，管理者无法判断线索状态、销售反馈和客户池变化。
3. 该页面把 PRD 中“少填、快看、可追溯”的要求落为具体操作。

### 2.3 页面目标（可衡量）

| 目标维度 | 具体指标 |
| :--- | :--- |
| 效率 | 关键操作在 3 次点击内完成 |
| 准确性 | 状态、负责人、来源和时间字段完整展示 |
| 可追溯 | 人工修改、导出、发送或提交均能回到记录 |

### 2.4 不做什么
- 不展示成交金额和报价金额，因为 PRD 明确排除金额类指标。
- 不做完整 CRM 的复杂商机阶段管理，第一版聚焦线索闭环。
- 不做未授权 Facebook/LinkedIn 抓取。

## 3. 用户与场景

### 3.1 目标用户

| 维度 | 描述 |
| :--- | :--- |
| 角色 | 管理员、营销/运营、销售，按权限看到不同动作 |
| 技术熟练度 | 中等，熟悉表格和基础后台操作 |
| 使用设备 | PC 浏览器为主 |
| 使用姿势 | 高频扫描、快速判断、少量确认 |

### 3.2 核心使用场景

**场景 A：正常处理**
- 时机：用户进入 客户态势信号记录 执行日常任务。
- 前置：已登录或持有有效安全链接。
- 动作：查看关键信息，完成筛选、确认、反馈、导入或查看详情。
- 情绪：清楚、稳定、知道下一步。

**场景 B：异常兜底**
- 时机：数据缺失、链接过期、权限不足或任务失败。
- 前置：当前流程无法自动完成。
- 动作：查看原因，进入待处理、重试、返回工作台或请求管理员处理。
- 情绪：不被卡住，知道谁来处理。

### 3.3 用户核心诉求
1. 快速看到状态和下一步。
2. 不重复录入系统已有信息。
3. 能追溯 AI 判断、销售反馈和人工修改来源。

## 4. 信息架构

### 4.1 区域划分
1. 顶部标题与关键动作。
2. 核心指标或摘要区。
3. 主操作区。
4. 明细表格、时间线或表单区。
5. 异常提示和权限说明区。

### 4.2 区域内容
- 标题区展示页面名称、阶段、当前状态和主操作。
- 主体区展示与 客户态势 相关的核心业务数据。
- 明细区展示客户、线索、反馈、报表或配置记录。

## 5. 界面内容

### 5.1 元素清单
- 页面标题：客户态势信号记录。
- 顶部指标：总信号数、待复核信号、官网公开信号、可进入再营销信号。
- 筛选控件：来源类型、状态、客户、页码和每页数量。
- 人工新增表单：客户、来源类型、标题、信号摘要、证据 URL、证据文本、可信度、状态。
- 记录列表：客户名称、国家、产品、信号来源、信号标题、摘要、可信度、状态、更新时间、查看客户详情。

### 5.2 辅助信息
- AI 判断理由、数据来源、更新时间、导出脱敏说明或链接有效期说明。

## 6. 交互流程

### 6.1 页面加载与初始化
- 校验登录态或安全链接。
- 拉取页面所需数据。
- 按业务时间倒序展示列表或记录。

### 6.2 核心操作流程

```text
进入客户态势信号记录
  -> 查看关键状态和上下文
  -> 执行主操作
  -> 系统保存结果并记录来源
  -> 跳转或回写到下游页面：客户详情
```

### 6.3 辅助操作
- 搜索、筛选、排序、重试、返回上级和导出确认。

## 7. 状态与边界

### 7.1 页面状态全景

| 状态 | 触发条件 | 页面表现 |
| :--- | :--- | :--- |
| 初始态 | 首次进入 | 展示标题、骨架或默认数据 |
| 加载态 | 请求中 | 骨架屏或按钮 loading |
| 完整态 | 数据可用 | 展示 mock 数据和关键动作 |
| 空态 | 无数据 | 给出导入、配置或返回入口 |
| 失败态 | 请求失败 | 展示原因和重试 |
| 无权限 | 角色不允许 | 隐藏或置灰动作并说明原因 |

### 7.2 边界与容错

| 边界场景 | 处理策略 |
| :--- | :--- |
| 网络异常 | 保留当前输入，允许重试 |
| 会话过期 | 跳回登录页或提示重新发送链接 |
| 连续错误 | 显示可理解原因，不吞掉错误 |
| 异常输入 | 当前字段就地提示 |
| 并发操作 | 以最新保存记录为准并保留操作人 |
| 权限不足 | 动作置灰或进入无权限页 |

## 8. 业务规则与校验

### 8.1 字段定义

| 字段 | 说明 |
| :--- | :--- |
| source_channel | 官网聊天、官网后台、邮箱、展会名片等来源 |
| country | 决定区域和销售负责人 |
| customer_type | 代理商、Doctor、Hospital、Clinic 等 |
| product_interest | Trolley、Portable、Handheld 或具体型号 |
| feedback_status | 已联系、已报价、无效、需跟进、已成交 |
| signal_source | website_public、email_interaction、sales_feedback、manual 四类来源 |
| signal_summary | 机会信号摘要，用于客户详情、再营销判断和后续人工复核 |
| evidence_url | 官网页面或授权归档链接；非公开/未授权社媒链接不得自动采集 |
| evidence_text | 信号证据摘录或人工说明 |
| confidence | 高、中、低或待复核 |
| signal_status | 待复核、已确认、可再营销、已归档 |

### 8.2 校验规则

| 校验项 | 时机 | 规则 | 失败反馈 |
| :--- | :--- | :--- | :--- |
| 必填项 | 提交时 | 关键字段不能为空 | 标记待补充 |
| 权限 | 操作前 | 角色必须允许动作 | 动作不可用 |
| 链接有效期 | 打开反馈卡片 | 7 天内有效 | 提示重新发送 |
| 导出字段 | 导出前 | 不包含成交金额和报价金额 | 阻止导出 |

### 8.3 特殊业务规则
- 销售判断优先于 AI 评分，但 AI 判断理由保留。
- 信号记录必须绑定客户详情页，动作入口进入 `/admin/customers/:customerId`。
- 人工新增或修改信号必须写入审计，保留 trace id、操作人、来源和目标客户。
- 给大模型或再营销使用信号时必须包裹为“数据上下文”，不得把网页/邮件/附件里的文本当成系统指令。
- 再营销必须人工确认后发送。
- 官网 KPI 只统计渠道归因和时效，不统计金额。

## 10. 数据需求

### 10.1 页面需要什么数据

| 数据 | 来源 | 用途 |
| :--- | :--- | :--- |
| 当前用户与角色 | Auth 模块 | 权限控制 |
| 客户态势信号 | CustomerSignal | 展示官网、邮件、销售反馈和人工录入的机会信号 |
| 线索/客户/反馈记录 | 后端业务模块 | 展示主体内容并生成默认信号 |
| 配置字典 | 系统配置 | 渲染国家、产品、销售选项 |
| 报表聚合 | ReportMetric | 展示指标 |

### 10.2 页面提交什么数据

| 数据 | 触发时机 | 用途 |
| :--- | :--- | :--- |
| 筛选条件 | 查询时 | 获取列表 |
| 人工信号 | 保存时 | 新增 CustomerSignal 并进入客户详情 |
| 状态或配置修改 | 保存时 | 更新业务记录 |
| 反馈或确认结果 | 提交时 | 写入闭环 |

### 10.3 明确排除的数据
- 成交金额、报价金额。
- 未授权社媒私域数据。
- 与当前页面无关的完整 CRM 字段。

## 12. 生产化落地补充（07-01-2026）

- 生产路由：`/admin/customer-signals`。
- 后端接口：`GET /api/customer-signals`、`POST /api/customer-signals`、`GET /api/customer-signals/context`。
- 数据模型：`CustomerSignal`，字段包括 `customer_id`、`signal_source`、`signal_title`、`signal_summary`、`evidence_url`、`evidence_text`、`confidence`、`status`、`observed_at`、`created_by`、`updated_at`。
- 列表必须分页，支持按 `source`、`status`、`customer_id` 筛选。
- 页面必须显示全局 Banner、指标卡、筛选区、人工新增信号表单和信号表格。
- 表格“查看客户详情”必须跳转到真实客户详情 `/admin/customers/:customerId`。
- 管理员和运营可访问、创建信号；销售角色访问页面/API 返回 403，并写入 `permission_denied` 审计。
- `GET /api/customer-signals/context` 返回 `CUSTOMER_SIGNAL_DATA_ONLY` 安全边界，供后续大模型再营销或客户洞察使用。
