# Ultrasound Growth 管理后台技术方案

## 1. 当前真实实现范围

本方案描述当前仓库已经落地的生产化实现，而不是静态原型或未来设想。阶段 1 MVP 的 18 个页面已接入 React 路由、FastAPI 接口与本地 SQLite 持久化；阶段 2 的再营销待办列表、草稿详情和客户态势信号记录已进入生产实现，用于覆盖提示词、附件素材、大模型草稿上下文、人工确认发送、客户机会信号记录和 AI 数据边界。

## 2. 技术栈

| 层级 | 当前实现 | 说明 |
|---|---|---|
| 前端 | React + Vite + TypeScript + Ant Design + lucide-react | 路由在 `frontend/src/main.tsx`，全局壳在 `frontend/src/shell/AppShell.tsx` |
| 后端 | FastAPI + SQLAlchemy + Pydantic | 入口为 `backend/app/main.py`，模型为 `backend/app/models.py`，契约为 `backend/app/schemas.py` |
| 持久化 | SQLite 本地开发数据库 | 当前测试和本地运行使用真实数据库读写；后续可按部署需要迁移到 Postgres |
| 文件上传 | FastAPI UploadFile + 数据库附件元数据 | 再营销附件当前保存文件名、类型、大小、上传人和时间，不把危险文件写入发送队列 |
| 审计 | `AuditLog` | 登录失败、权限拒绝、配置变更、导出、客户背景修订、再营销保存/附件/重新生成/确认、客户态势信号新增均写审计 |
| 认证权限 | JWT + RBAC | 管理员/运营可访问后台配置、报表、再营销和客户态势；销售角色受后端 403 和前端路由守卫限制 |

## 5. 再营销生产化设计

再营销不再是客户池页里的只读草稿。当前实现为独立任务流：

1. `GET /api/nurture-tasks` 分页读取 NurtureTask，列表展示客户、分层、建议下一步动作、客户备注、提示词/附件状态和草稿状态。
2. `GET /api/nurture-tasks/{id}` 打开草稿详情，返回客户摘要、背景调查、销售反馈、附件和提示词上下文快照。
3. `PUT /api/nurture-tasks/{id}` 保存建议下一步动作、客户备注、触达理由、草稿正文和人工提示词，并写 `nurture_task_updated` 审计。
4. `POST /api/nurture-tasks/{id}/attachments` 上传合规附件素材，拒绝危险扩展名和超大文件，写 `nurture_attachment_uploaded` 审计。
5. `POST /api/nurture-tasks/{id}/regenerate` 使用客户摘要、背景调查、销售反馈、提示词和附件元数据生成新草稿，记录 `model_provider=ultrasound_growth_llm` 与 `model_version=nurture-draft-v1`。
6. `POST /api/nurture-tasks/{id}/confirm` 将草稿改为 `confirmed`，同一任务重复确认不重复写副作用，写 `nurture_task_confirmed` 审计。
7. 销售角色访问再营销接口返回 403，权限拒绝写 `permission_denied` 审计。

LLM 上下文使用 `<customer_context>` 分隔符，并声明其中内容为数据而非指令，避免客户输入、网页内容或附件名称中的 prompt injection 被当成系统指令。

## 6. 客户态势信号生产化设计

客户态势信号记录不再是静态演示页。当前实现为独立的管理员/运营工作台：

1. `GET /api/customer-signals` 分页读取 CustomerSignal，支持按来源、状态、客户筛选，返回总信号、待复核、官网公开信号和可再营销信号四类摘要。
2. `POST /api/customer-signals` 人工新增信号，来源限定为 `website_public`、`email_interaction`、`sales_feedback`、`manual`，非法 `facebook_scrape` 等未授权抓取来源由 Pydantic 拒绝。
3. 每条信号返回 `customer_detail_path=/admin/customers/:customerId`，前端表格动作直接进入真实客户详情。
4. `GET /api/customer-signals/context` 输出 `CUSTOMER_SIGNAL_DATA_ONLY`，使用 `<customer_signals>` 分隔符并声明网页、邮件或人工录入内容是数据，不是系统指令。
5. 管理员/运营可访问和创建信号；销售角色访问客户态势接口返回 403，并由 `require_admin_or_ops` 写入 `permission_denied` 审计。
6. 默认种子信号来自 GlobalMed Peru 官网公开信息、授权邮件互动和销售反馈，便于本地环境与测试环境均可验收。

## 7. 工程红线落实

| 红线 | 当前落实 |
|---|---|
| 分页查询 | 线索、客户、报表、设置子页、再营销任务、客户态势信号均使用 page/page_size 或 limit |
| 权限 | `current_user` + `require_admin_or_ops`，销售角色后端禁访配置、报表、分配、再营销、客户态势 |
| 审计与 trace | 中间件生成 `x-trace-id`，关键操作写 `AuditLog` |
| 幂等 | 销售反馈链接一链一反馈；再营销确认重复提交不重复写确认审计 |
| 敏感信息保护 | 导出脱敏；过期/无效反馈链接不展示客户详情 |
| LLM 隔离 | 产品知识、再营销上下文和客户态势信号上下文均用分隔符声明“数据非指令” |
| 附件安全 | 再营销附件限制扩展名和大小，失败不破坏已有草稿 |
| HTML 原型口径 | `output/pages/*.html` 仅作为视觉、布局和组件蓝图；生产行为以 React + FastAPI 为准 |

## 9. 已知后续项

- Vite 首包仍大于 500k，后续可用动态 import 和 manualChunks 拆分。
- FastAPI `on_event`、`datetime.utcnow` 有弃用 warning，功能不阻塞，但后续应迁移到 lifespan 和 timezone-aware datetime。
- 当前本地持久化使用 SQLite；正式部署如接 Postgres/Redis/对象存储，需要补部署配置和迁移脚本。
